1The 60-second TL;DR

PPP is getting a second operating platform, built from scratch, living at the same hub URL as the existing Command Center. Where the Command Center mirrors Salesforce, the New Platform is the system of record. Every account, bid, project, invoice, change order, daily report, and closeout document lives in Postgres — owned by the app, not synced from somewhere else.

  • Same login. One sign-in at hub.precisionpaintingplus.net → picker → either platform.
  • One sidebar switcher at the bottom of each platform's nav for users with access to both.
  • Nine phases from your operating-system diagram, each its own build slice with schema, screens, and notifications.
  • Postgres-native means edits land instantly, no 30-min SF sync delay, and we don't depend on SF schema for new fields.
  • Strict backend separation — code, database tables, sidebars, notifications, and invoices are all on parallel tracks. The two platforms cannot interfere with each other.
  • Phase 0 (foundation, login picker, RBAC, audit log) is shipped. Phase 1 (Account Management) starts next.

2How it lives alongside Command Center

Both platforms share one URL, one login, and a thin shell layer. Everything else is parallel.

The thin shared layer (only these touch both)

  • Auth — Supabase + Google SSO + the profiles table with per-platform access flags.
  • Shell components — topbar primitives, sidebar primitives, user menu.
  • Notification bell — one component, but notification kinds are platform-prefixed (cc.* vs np.*).
  • Document storage — Supabase Storage (separate buckets per platform).
  • Resend (transactional email) — shared client, different templates.
  • Mobile design tokens — 44px tap targets, safe-area-inset, color palette.
Cross-imports are banned. A linter rule + a lib/commercial/README.md codify that lib/salesforce/* and lib/commercial/* never import from each other. The audit on the day Phase 0 shipped confirmed zero violations.

The 12 statuses (DAG — server-enforced transitions)

Lead → Qualified → Site Visit Required → Site Visit Complete → Plans/Specs Received → Estimating → Internal Review → Proposal Submitted → Follow Up → Negotiation → Verbal Award → Contract Review → Awarded | Lost | No Bid

Inbound bid paths

  • Email — bids@[PPP domain] inbox. Inbound webhook parses sender + subject + body + attachments → opens a new opp in "Lead" status. Auto-links to Account if the sender's domain matches.
  • Online form — public token-gated form. PPP sends a link to a known prospect; submission lands as an opp + attaches the prospect's contact info.
  • Manual — estimator types it in (referral, phone call, repeat client).

Edge cases handled

  • Inbound from unknown sender → opp created with placeholder Account + "needs linking" flag.
  • "Awarded" status flip auto-creates a commercial_projects row (Phase 5).
  • "Lost" requires lost_reason + lost_to (for win/loss reporting).
  • Estimator scope — by default only their own opps; admin sees all.
  • Salesperson reassignment audit-logged + bell-notified.

9Document storage

Supabase Storage. Separate buckets per platform so a leak in one can't expose the other.

  • Bucket name: commercial-documents
  • Path scheme: /{account_id}/{project_id?}/{category}/{filename}
  • Access: signed URLs only, 1-hour TTL.
  • Versioning: immutable storage keys; new uploads bump version_n in the row.
  • Retention: 7 years (construction industry norm), longer for warranty docs.
  • Upload size: chunked, supports up to 500MB per file (large architectural PDFs).
  • Compression: images server-side resize to a 2K thumbnail + keep original.
  • Search: filename + category + uploaded-by; full-text on extracted PDF text (Phase 6.5).
